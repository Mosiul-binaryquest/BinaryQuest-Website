Our Delivery Process

How every BinaryQuest engagement actually runs

Regardless of which collaboration model you choose, every BinaryQuest project follows the same governance structure — adapted to your model's rhythm. Inspired by how Thoughtworks, GitLab, and Atlassian structure distributed team delivery.

1

Discovery & Scoping (Week 1–2)

We start every engagement with a structured discovery phase — the #1 thing that separates successful from failed software projects. We map your requirements, constraints, data landscape, and business objectives. We challenge assumptions, surface hidden complexity, and produce a scoping document with acceptance criteria, effort estimates, and a recommended collaboration model.

This phase mirrors the UK Government Digital Service's discovery methodology and Accenture's client alignment process. For Fixed Price projects, nothing moves to build without a signed scope. For T&M or Dedicated Team, this becomes your backlog foundation.

Stakeholder interviews Requirements mapping Acceptance criteria Effort estimation Risk register Collaboration model recommendation
2

Sprint Execution (2-week cycles — industry standard)

We run 2-week sprints — used by 59% of agile teams globally and endorsed by Atlassian, GitLab, and Thoughtworks as the optimal cadence balancing speed with stability. Every sprint starts with sprint planning (scope + commitment), includes a daily 15-minute stand-up (video for distributed teams, async log for timezone-split teams), and ends with a demo and retrospective.

Teams of 10 or fewer per squad (Scrum Guide recommendation) — small enough for genuine daily communication, large enough for meaningful parallel delivery. Velocity tracked from day one; we use actual throughput for planning, not optimistic estimates.

2-week sprints Daily 15-min stand-up Sprint planning Sprint demo Retrospective Velocity tracking
3

Weekly Client Communication

Every Friday, your account manager sends a written status report: what was delivered this sprint, what's planned for next, velocity trend, open risks, and any decisions needed from you. No surprises on Monday. This mirrors Basecamp's philosophy of "no one should have to ask what's happening" — the information comes to you, not the other way around.

You also have live access to our GitHub/GitLab repository and Jira board — complete visibility into every commit, pull request, and backlog item without having to ask. Thoughtworks calls this "transparency as a delivery practice."

Friday written status report Live GitHub/GitLab access Live Jira/project board Risk log updated weekly Decision log
4

Monthly Steering Review (for retainer and long-term engagements)

A 60-minute formal review with your stakeholders: programme-level progress, KPI dashboard, financial tracking (hours used vs. budget remaining for T&M, milestone progress for Fixed Price), risk and issue status, and upcoming priorities for the next month. Decision-makers can redirect priorities at this meeting with immediate effect on the next sprint.

This governance layer mirrors SAFe (Scaled Agile Framework) as used by Accenture across their Global Delivery Network — enterprise-grade oversight scaled to mid-market reality.

60-minute formal review KPI dashboard Budget tracking Risk status Priority realignment Escalation path
5

Delivery, Handover & Knowledge Transfer

At every milestone and at project close, BinaryQuest performs a structured handover: all source code committed to your repository, environment configuration documented, credentials transferred, runbooks written, and a recorded walkthrough of the system for your team. If the engagement ends, you can run without us from day one.

For Dedicated Team and BOT engagements, knowledge transfer is continuous — documentation is updated in every sprint, not at the end. Institutional knowledge never lives only in the heads of BinaryQuest developers.

Source code handover Environment docs Credential transfer Recorded walkthroughs Runbooks Continuous documentation

Tools & Practices

How we collaborate — tools and practices used

Inspired by GitLab's all-remote model, Basecamp's async-first philosophy, Atlassian's agile frameworks, and Thoughtworks' distributed team practices — adapted for the South Asian market and time zone.

🔄 Async-First Communication

  • Written first: Decisions, updates, and status changes documented in Jira/Notion before being communicated verbally — so nothing is lost in a call.
  • Loom video updates: Complex explanations shared as short recorded videos rather than scheduling a meeting — you watch on your time.
  • Shared decision log: Every significant decision recorded with the rationale — no one has to remember what was agreed in a meeting 3 months ago.
  • Timezone-aware standups: For global teams, async daily log replaces synchronous standup when timezone overlap is less than 4 hours — inspired by GitLab's all-remote handbook.
  • No-surprise principle: Inspired by Basecamp — the team pushes information to you, you never have to pull it.

🛠️ Engineering Transparency

  • GitHub / GitLab access: Client has read access to repository from day one — every commit, pull request, and code review is visible.
  • CI/CD pipeline: Automated build, test, and deploy pipeline — you can see every deployment and its status in real time.
  • Jira / Linear live board: Sprint board, backlog, and velocity chart shared with client — no need to ask "what's the team working on?"
  • Code review culture: All code peer-reviewed before merge. AI-assisted code review (GitHub Copilot-style tools) used internally — 55% faster delivery per Microsoft research.
  • Security scanning: Automated dependency vulnerability scanning (Snyk/Dependabot) on every repository — issues flagged before they reach production.

📋 Project Governance

  • Risk register: Maintained and shared weekly — every identified risk has a likelihood, impact, and mitigation owner.
  • Change log: Every scope change formally logged with client sign-off before work begins — eliminates "I didn't know that was extra" disputes.
  • Budget tracker: For T&M and Retainer clients — live hours consumed vs. budget, updated weekly. Never a surprise invoice.
  • Retrospective actions: Every sprint retrospective produces a list of improvements — tracked to completion in the next sprint. We get better every iteration.
  • Escalation path: Delivery manager → Account manager → Director. Clear, fast escalation with 4-hour response SLA at every level.

🔧 Standard Tool Stack

  • Project management: Jira (enterprise), Linear (startups), or Notion — client preference honoured.
  • Version control: GitHub or GitLab — client repository ownership from day one.
  • Communication: Slack (preferred), Microsoft Teams, or WhatsApp Business for South Asian clients.
  • Documentation: Confluence, Notion, or GitHub Wiki — maintained throughout the engagement, not just at the end.
  • Design collaboration: Figma with client editing access — you can comment on designs before a line of code is written.
  • CI/CD: GitHub Actions, GitLab CI, or AWS CodePipeline depending on stack.
